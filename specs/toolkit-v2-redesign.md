# Feature: Toolkit V2 Redesign — Plug-and-Play Verification Infrastructure

> Generated by `/feature` on 2026-03-27. Review before implementation.

---

## Intent

### 1. What problem does this solve?

Engineering leaders under executive pressure to adopt AI have no plug-and-play system that gives them the best odds of success — the right gates, the right policy, the right processes — without requiring months of custom instrumentation or expensive SaaS.

They currently default to: give developers Copilot/Cursor, measure "adoption rate," report "productivity up 40%," and get blindsided 3 months later when rework, review bottlenecks, and escaped defects eat the gains. They're optimizing a non-bottleneck (code generation speed) while verification — the actual constraint — goes unaddressed.

### 2. How will we know it worked?

**Level 1 — Adoption (measurable by the tool itself):**
- Team went from no AI policy to a defensible one they can present to leadership
- Team went from zero automated gates to at least Tier 0 blocking in CI
- Team has a weekly review running with basic metrics

**Level 2 — Impact (measurable only with ecosystem instrumentation):**
- Machine catch rate > 0% via CI reporting (guidance provided, not custom scripts)
- The full picture (rework rate, escape rate, cost per accepted change) requires Jira + CI/CD + review tooling that varies by team. We provide guidance and point to established tools (DORA ecosystem: Swarmia, LinearB, Sleuth, etc.), not custom measurement scripts.

### 3. What is out of scope?

- Plug-and-play for all CI/CD ecosystems — GitHub Actions guidance provided, others get general guidance not configs
- AI code detection / classification
- Individual developer productivity measurement
- Ongoing monitoring / dashboards — it's a setup tool + periodic manual review, not a SaaS
- Replacing commercial tools — recommend best free/open-source, acknowledge where commercial is better, but that's a later conversation
- Custom measurement scripts — heavily overlap with DORA, direct users to established tools
- Solving organizational change management — the tool provides gates and framing, the leader manages the humans

### 4. What must NOT happen?

**False confidence.** The tool must never imply that installation = safety. Loud warnings throughout: "This is a starting point, not a finish line." Every output should reinforce this.

**Gate theater.** The tool must be explicit that Tier 0 gates catch the cheapest failures only. "If your machine catch rate is only catching lint violations, your gates are catching nothing that matters."

**Claiming to measure what it can't.** No fake dashboards, no vanity metrics, no "your delivery health score is 87%." If we can't reliably measure it from git alone, we say so and point to what they'd need.

**Legally dangerous policy guidance.** The AI policy template must prompt for regulatory requirements (HIPAA, SOC2, FedRAMP) and include a disclaimer: "Have your legal/compliance team review before adopting."

### 5. Pre-mortem findings

1. **Measurement framework gets superseded.** DORA evolved, this might too. Mitigation: gates and policy are valuable independently of the measurement theory. Don't couple them.
2. **Gates become evidence against AI.** Teams use gate failures as ammunition to kill AI adoption. Mitigation: frame catches as value. "Your gates caught 47 issues this month that would have shipped to production" is a success story.
3. **Time-to-first-value too high.** If it's not easy, leaders under pressure will bounce. Mitigation: AI-guided setup gets them to Level 1 in under an hour. The detailed guides are reference material, not prerequisites.
4. **Stack mismatch.** Real, accepted, open to contributions. Not solvable for free on day one. AI can web-search for unfamiliar stacks as a fallback.
5. **The by-language recommendations will rot.** Tools get abandoned, acquired, or go paid. Mitigation: "last verified" dates on recommendations. AI should check tools still exist before recommending install.

## Behavioral Spec

### User Stories

**Current experience:** Leader gets mandate to adopt AI → tells team to use it → measures adoption rate → reports "productivity up" → 3 months later velocity is flat or worse, no data to explain why, no infrastructure to fix it.

**Desired experience:** Leader gets mandate → finds toolkit → opens repo with AI coding tool → AI detects their stack, audits what gates exist, recommends what's missing → leader gets: (1) defensible AI policy with reasoning, (2) Tier 0 gates blocking in CI, (3) basic weekly review format → can report outcomes not activity in next leadership meeting → has breadcrumbs for going deeper.

The setup process itself is AI-assisted, which means the leader can truthfully say "we used AI to set up our AI verification infrastructure."

### Mechanism

Curated defaults reduce decision fatigue. Gates catch real issues automatically. Data exists for the leader to report on. The leader reports outcomes (catches, review quality, trend direction) instead of activity (adoption percentage).

The causal chain: opinionated toolkit → AI-guided setup → gates in CI → catches happen → leader has evidence → executive pressure satisfied with substance.

### States and Transitions

| State | Description |
|-------|-------------|
| **Nothing** | No policy, no gates, no AI process |
| **Installed** | AI policy exists, Tier 0 gates blocking in CI, context files in repo, weekly review format ready |
| **Running** | Weekly review happening, gates catching things, leader has data to report |
| **Wired up** | CI reporting connected, DORA tools in place, ecosystem connectors for deeper metrics |

- Nothing → Installed: AI-guided interactive setup (CLAUDE.md / AGENTS.md)
- Installed → Running: team uses it for a few weeks (human behavior, not tooling)
- Running → Wired up: optional, for teams that want deeper measurement

### Error Cases

**Conflicting existing setup.** Most teams aren't at zero. The AI detects what exists and fills gaps. It asks before changing anything: "I see you have ESLint but no secret detection. Want me to add TruffleHog?"

**Unknown language/stack.** AI doesn't have a curated quick-start. Fallback: AI web-searches for best-in-class tools for that language, warns the user it's a search not a curated recommendation, builds the config together with the user.

**AI generates a bad CI config.** Every generated artifact is saved to `.deliverygap/rollback/`. AI asks the user how to test safely: "Can I push to a branch? Is there a dry-run? Should I just show you?" If the test fails, AI offers to revert from rollback.

**Multi-repo orgs.** Per-repo. Leaders can delegate setup to team leads. No org-level aggregation on day one.

**Gate noise.** Curated, opinionated defaults mitigate this. The guides recommend specific tools and configs, not "install everything." If existing codebase has thousands of warnings, guidance says: set baseline so only new violations block.

## Design Approach

### Conceptual approach

**B+C: Opinionated guides + AI-guided setup.**

The markdown guides (quick-starts, policy template, measurement guidance) are the source of truth. The CLAUDE.md / AGENTS.md file in the repo instructs the AI to read these guides and apply them interactively to the user's repo. The AI is the install experience. The markdown is the knowledge base.

No custom CLI to build or maintain. Works with Claude Code, Cursor, Codex, Goose, or any tool that reads context files.

### Codebase context

The toolkit today is 44 markdown files, 5 Python scripts (being killed), and 1 installable Python package (multi-pass review). It's a reference library that reads like documentation, not a setup tool. The redesign restructures it into an opinionated, interactive experience while keeping the reference material accessible for teams that want to go deeper.

### Structural issues flagged

- **`delivery-metrics/` must be deleted entirely.** Scripts, READMEs, best practices docs — all of it. Replaced by `measurement-guidance/` which points to established tools.
- **`by-language/` pages need trimming.** From 13+ tools per tier to 3-4 opinionated picks with "last verified" dates.
- **Templates reference dead scripts.** Rollout memo, scorecard references, spec best practices all reference measurement scripts being killed. Must be updated.
- **No CLAUDE.md exists.** This is the single most important file in the redesign. It's the entry point for the AI-guided setup experience.
- **Quick-starts describe what should be in place, not copy-paste configs.** V1 says what Tier 0 includes for each stack and what tools to use. Copy-paste CI workflow files are a future improvement once tested against real repos.

### Constraints and Non-negotiables

- **Security:** CLAUDE.md must NOT instruct the AI to run arbitrary commands. It generates configs, the human reviews and applies. All generated artifacts reference specific files in the repo, not AI memory. Rollback checkpoints saved for every generated artifact.
- **Reliability:** The AI-guided setup must degrade gracefully. If the AI can't detect the stack, it asks. If there's no quick-start for the language, it searches and warns. If it can't search, it points to the by-language reference pages.
- **Compliance:** AI policy template prompts for regulatory requirements and includes "have legal review" disclaimer.

### Known AI Blind Spots

- [x] Edge cases: Unknown language/stack → web search fallback with warning. Existing warnings in codebase → guidance on baselining.
- [N/A] Concurrency: Setup tool, not a service.
- [x] Error handling: Bad CI config → rollback checkpoint + user-guided safe testing. AI asks how to test, doesn't assume.
- [x] Security: CLAUDE.md doesn't authorize arbitrary command execution. Generated configs are reviewed by human. Policy template prompts for regulatory requirements.
- [x] Non-prompted concerns: Tool recommendation rot (mitigated by "last verified" dates and AI checking tool existence). Multi-pass review requires API key (disclosed upfront, alternatives listed).
- [x] Hallucination risk: CLAUDE.md references specific files in repo. AI reads guides, doesn't generate from memory. Web search results are flagged as unverified.

### Rollback Plan

- **Trigger signal:** Users report "setup broke my CI" or "policy template caused compliance issues"
- **Method:** Git revert on bad configs. Per-setup rollback via `.deliverygap/rollback/`. Policy template is guidance — always reviewable by legal.
