# Ship

Create a pull request after /build completes, auto-populating the PR description from the feature spec so reviewers get full context without reading the spec themselves.

## Input

Optional argument: `$ARGUMENTS` (path to a spec file).

If no spec path is provided, look in the `specs/` directory for the most recently modified `.md` spec file (exclude plan, progress, and DECISIONS files). If multiple specs exist and it is ambiguous, ask the user which one.

## Precondition Checks

Before creating the PR, verify:

1. **Uncommitted changes**: Run `git status`. If there are uncommitted changes, warn the user and ask whether to commit first. Do not proceed until the working tree is clean or the user explicitly says to continue.

2. **Branch check**: Determine the current branch. If on `main` or `master`, ask the user whether to create a feature branch first or PR from the current branch. Do not silently PR from main.

## Extract Spec Context

Read the spec file and extract these sections. Match by heading content, not exact formatting:

- **Intent / Problem**: The problem being solved and any success metrics. Look for headings like "Intent", "Problem", "Why", or the opening section.
- **Behavioral spec summary**: What the feature does. Look for "Behavioral Spec", "Behavior", "What it does", or similar.
- **Architectural decisions**: Key design choices and rationale. Look for "Architecture", "Design", "Decisions", or similar.
- **Constraints / Non-negotiables**: What must NOT happen, boundaries. Look for "Constraints", "Non-negotiables", "Boundaries", or similar.

If `specs/DECISIONS.md` exists, read it and extract any decisions related to this feature (match by feature name or keywords from the spec).

## Gather Commits

Determine the feature name from the spec filename (e.g., `specs/auth-flow.md` -> `auth-flow`).

Run `git log --oneline` and collect commits matching the pattern `feat(feature-name)` or similar conventional commit prefixes for this feature. Use these to understand what was built.

Also check for a progress file at `specs/[feature-name]-progress.md` to reference in the verification section.

## Create the PR

Use `gh pr create` with a structured description. Derive the title from the spec's problem statement -- keep it under 70 characters, concise, and descriptive.

PR body format:

```
## Why
[Problem being solved, from spec's intent section]

## What
[Behavioral summary -- what the feature does, from spec's behavioral spec]

## Key decisions
[2-3 most important design decisions and why, from spec and DECISIONS.md]

## Constraints
[Non-negotiables from the spec -- what must NOT happen]

## Verification
- [x] All phases completed (link to progress file if it exists)
- [x] Red team review passed
- [x] Learnings captured
- [ ] Manual verification by reviewer: [list items from spec's acceptance criteria or verification section]

## Spec
Full spec: `specs/[feature-name].md`

---
Generated by Upfront /ship
```

Use a HEREDOC to pass the body to `gh pr create` for correct formatting:

```bash
gh pr create --title "the pr title" --body "$(cat <<'EOF'
## Why
...

---
Generated by Upfront /ship
EOF
)"
```

## After PR Creation

Report the PR URL to the user. If `gh pr create` fails, show the error and suggest fixes (e.g., push the branch first, set upstream).
